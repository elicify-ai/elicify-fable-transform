From fablize to elicify-fable-transform

A side-by-side walkthrough of how each harness injects instructions into the model, what triggers the injection, and what makes the model actually behave like Fable.

Plain English. Mermaid diagrams. The original Claude plugin on the left, the new opencode plugin on the right.

1. The 30-second version

Original Claude plugin fablize

A Claude Code plugin that runs shell/Python hooks on three Claude events: every user prompt, every tool call, and when the LLM tries to stop. The hooks write a small JSON blob that Claude's runtime injects as a user or system message on the model's next turn.

New opencode plugin elicify-fable-transform

An opencode plugin that wires two transform hooks on the LLM call: experimental.chat.system.transform (per-session, recommended) and experimental.chat.messages.transform (global, fallback). Other code calls enqueue(sessionID, directive) to schedule a directive for the next LLM turn.

Same goal, different mechanic. Both harnesses try to put a "verification reminder / stop-block reason" into the model's input on the next turn. The Claude plugin does it by stamping a tool's output (with a flag Claude reads as additionalContext). The opencode plugin does it by appending a tagged block to the system prompt at LLM-call time.

4. The single source of truth

"Source of truth" here means: where is the contract that says "the model must verify before it claims done"?

fablize — two sources, both true

  1. Static (always-on): setup/setup.sh injects fablize-block.md into ~/.claude/CLAUDE.md. The model sees it on every turn as a system instruction.
  2. Dynamic (per-event): hooks/hooks.json + the four gate_*.py scripts. The runtime calls them on UserPromptSubmit / PostToolUse / Stop and their additionalContext lands as a real user message in the next turn.

The two are consistent because the static block is the long-form contract and the dynamic hooks are the short, contextual nudges.

elicify-fable-transform — three layers, one contract

  1. Static (always-on): the elicify-fable-architect.md agent's <elicify_fable_operating_mode> block. It is the model-resident operating procedure.
  2. Skill (on demand): /elicify-fable-transform slash command (SKILL.md) is invoked by the user or auto-loaded by the agent when steering is needed.
  3. Dynamic (per-session): the plugin's DirectiveQueue, drained on every LLM call by experimental.chat.system.transform, appending <elicify-fable-directives>…</elicify-fable-directives> to the system prompt.

All three converge on the same wording of the verification reminder, so the model cannot tell whether it came from the system prompt, the skill, or the runtime queue — and that is the point.

6. Anatomy of an injection

What does the model actually see in its input? Here is the literal text.

fablize (from gate_prompt.py)

// Returned in hookSpecificOutput.additionalContext
[elicify-fable] Verification reminder: before reporting a task as done,
- observe the actual output of the change (run the test, render the artifact, hit the endpoint);
- ground any "done" claim in a tool result from this turn, not in intent;
- if a step failed and you cannot fix it, surface that explicitly.

Claude's runtime treats this as a real user turn on the next LLM call. It is a strong steer because the model reads it as a fresh message from a participant.

elicify-fable-transform (appended to output.system)

<elicify-fable-directives ts="2026-07-23T11:24:00Z">
[elicify-fable:contract @ 2026-07-23T11:24:00Z]
[elicify-fable] Verification reminder: before reporting a task as done,
- observe the actual output of the change ...
- ground any "done" claim in a tool result from this turn ...
- if a step failed and you cannot fix it, surface that explicitly.
</elicify-fable-directives>

opencode's runtime appends this to the system prompt. The XML-style wrapper is not a magic tag — it is a visual marker so the model (and you, when reading logs) can recognise harness-injected text. The LLM treats it as a system instruction: a higher-priority steer than user-typed text.

7. What makes the model behave like Fable

Both harnesses converge on a small set of behavioural instructions, just delivered in different ways. Here is what is in the contract, regardless of which harness delivers it:

BehaviourHow it is conveyed in fablizeHow it is conveyed in elicify-fable-transform
Verify before reporting done Injected as additionalContext on every prompt + as the always-on block in ~/.claude/CLAUDE.md. In the architect agent's <reasoning_protocol> step 7 + as the default elicify-fable:contract directive in the plugin's system prompt.
Refuse to finish unverified Stop hook returns decision: block with a reason. Claude re-prompts the model. event(session.idle) handler calls client.session.prompt() to inject a re-prompt turn, and/or enqueues a stop:block directive that lands on the next LLM call.
Don't retry silently PostToolUse detects a repeated failure class and returns additionalContext "stop retrying silently". Same logic can call t.enqueue(sessionID, { id: "post-tool:repeat-failure", text: "..." }); the directive lands on the next LLM call.
Escalate at the capability ceiling Static block in CLAUDE.md & in the /fablize SKILL.md. Static block in the elicify-fable-architect.md agent's <uncertainty> section.
Use the goal engine on multi-step work Referenced in CLAUDE.md block and SKILL.md. Lives at ~/.config/opencode/fablize/scripts/goals.py. Referenced in the architect agent's <goals> section. Same script (the elicify-fable plugin reuses it via the existing install).

8. What is correct about each, what is wrong

fablize

Correct: additionalContext is genuinely a separate message in Claude's next turn, so steering is strong.

Also correct: the Stop hook can block, which is a real Claude control-plane action, not a polite suggestion.

Limitation: the harness is tied to Claude's plugin runtime — it cannot be ported verbatim to a non-Claude runtime.

elicify-fable-transform

Correct: system.transform is the SDK-native way to mutate the system prompt. It is the right hook for "inject a directive on the next LLM call".

Also correct: the per-session FIFO queue gives any other hook (Stop, PostToolUse, custom event) a clean way to schedule a directive without coupling.

Known gap: the SDK's messages.transform does not yet expose sessionID, so the global drain is intentionally lossy — prefer system.transform.

9. What the opencode plugin fixes (vs. a naive port)

A naive port of fablize into opencode would do this in tool.execute.after:

// NAIVE — what you should NOT do
async "tool.execute.after"(input, output) {
  const ctx = await runGatePostTool(input, output)
  if (ctx) output.output = `${output.output}\n\n${ctx}`   // ❌ stamps tool output
}

It reaches the LLM (the stamped text is in the tool result on the next turn) but the LLM reads it as the tool's reply, not as a directive. Steering is weak.

What elicify-fable-transform does instead:

// CORRECT — what the plugin actually does
async "experimental.chat.system.transform"(input, output) {
  const queued = queue.drain(input.sessionID)
  const block  = formatDirectives(queued)   // tagged 
  if (block) output.system = [...output.system, block]
}

Now the text is on the system channel, which the model treats as a higher-priority instruction than tool output. Same intent, correct layer.

10. One-page summary

flowchart LR subgraph CLAUDE[fablize on Claude] direction TB UPS[UserPromptSubmit]:::claude --> RI[router.sh + gate_prompt.py]:::claude --> AC1[additionalContext]:::claude PTU[PostToolUse]:::claude --> GPT[gate_post_tool.py]:::claude --> AC2[additionalContext]:::claude STP[Stop]:::claude --> GST[gate_stop.py]:::claude --> BLK[decision block]:::claude end subgraph OC[elicify-fable-transform on opencode] direction TB ENQ[enqueue sessionID,id,text]:::oc --> Q[DirectiveQueue]:::oc ST[chat.system.transform]:::oc --> DR[drain sessionID]:::oc --> APP[append to output.system]:::oc MT[chat.messages.transform fallback]:::oc --> DR2[drain all]:::oc --> SYN[synthetic part on last msg]:::oc IDLE[event session.idle]:::oc --> CL[client.session.prompt]:::oc end classDef claude fill:#fef3c7,stroke:#d97706,stroke-width:1px,color:#1a1a1a classDef oc fill:#dbeafe,stroke:#2563eb,stroke-width:1px,color:#1a1a1a
  • Same intent: put a verification reminder / stop-block reason into the LLM's next-turn input.
  • Different mechanism: Claude = additionalContext returned from a hook, injected as a user message. opencode = system.transform appends to the system prompt; the plugin exposes a tiny enqueue() for hooks to use.
  • Source of truth: in both cases it is the always-on block (CLAUDE.md / architect agent system prompt) plus the dynamic hooks. The two are kept in sync by deliberate wording, not by code generation.
  • Triggers: per-event hooks (Claude) vs. per-LLM-call transform + per-event event() handler (opencode). The opencode model is more composable because enqueue() decouples the producer from the consumer.
  • Common pitfall: stamping output.output in a tool hook is the wrong layer in both runtimes — it reaches the model as the tool's reply, not as an instruction.
